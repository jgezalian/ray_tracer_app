Ballistics app deployment checklist (EC2 + Docker backend + local Postgres + nginx + S3)

This checklist is based on the current app snapshot and is tailored to the code/config patterns in the repo:
- Angular frontend in ballistics_frontend
- Spring Boot backend in ballistics_backend/backend
- Current compose file in ballistics_backend/compose.yaml
- Postgres currently containerized in compose, but this checklist assumes Postgres will run on the EC2 host to reduce cost
- S3 already used by the backend for rendered video storage

==================================================
1. Target production architecture
==================================================

Use this shape for the first EC2 deployment:

Internet
  -> nginx on EC2 (ports 80/443)
     -> serves Angular static build at /
     -> proxies /api/* to Spring Boot backend container on localhost:8080
     -> optionally proxies OAuth endpoints to backend too
  -> Postgres running locally on EC2, not exposed publicly
  -> backend runs in Docker and talks to Postgres over the private/local network
  -> rendered videos stored in S3

Recommended first-pass services:
- EC2 instance: t3.small minimum, t3.medium preferred
- EBS volume for app + DB data
- S3 bucket for outputs
- IAM role attached to EC2 for S3 access
- Route 53 + domain optional
- SSL via Let's Encrypt or AWS-managed option behind a load balancer if added later

==================================================
2. Biggest issues to fix before deploying
==================================================

The current codebase still has local-development assumptions that will break on EC2 unless changed.

2.1 Remove hardcoded localhost frontend origin

Found in backend source:
- backend/src/main/java/com/jgezalian/ballistics_backend/config/OAuth2Login.java
- backend/src/main/java/com/jgezalian/ballistics_backend/controllers/JobsController.java
- backend/src/main/java/com/jgezalian/ballistics_backend/controllers/TestController.java
- backend/src/main/java/com/jgezalian/ballistics_backend/controllers/UserController.java

Current values include:
- http://localhost:4200
- defaultSuccessUrl("http://localhost:4200", true)
- logoutSuccessUrl("http://localhost:4200")

Action:
- Replace all hardcoded frontend URLs/origins with environment variables.
- Example variables:
  APP_FRONTEND_URL=https://your-domain.com
  APP_ALLOWED_ORIGIN=https://your-domain.com
- Centralize CORS config instead of using repeated @CrossOrigin annotations with localhost.

2.2 Replace frontend dev proxy assumptions

Current frontend file:
- ballistics_frontend/proxy.conf.json

Current target:
- http://localhost:8080

This is only for local ng serve. In production, nginx should proxy /api to the backend.

Action:
- Keep proxy.conf.json for local development only.
- In production frontend code, use relative API paths like /api/... instead of hardcoded localhost backend URLs.
- Verify Angular code does not hardcode localhost anywhere else.

2.3 Secure secrets immediately

Current root .env contains real-looking credentials and weak local values:
- GOOGLE_CLIENT_ID
- GOOGLE_CLIENT_SECRET
- POSTGRES_PASSWORD=password123
- login_url=http://localhost:8080/oauth2/authorization/google

Action:
- Treat current secrets as compromised if they were ever committed or shared.
- Rotate Google OAuth credentials.
- Use a production .env on EC2 that is not committed.
- Use a strong database password.
- Do not keep production secrets in the repo or zip exports.

2.4 Fix backend DB connection shape

Current backend resources include:
- application.properties using jdbc:postgresql://db:5432/${POSTGRES_DB:ballistics_db}
- application-dev.properties using jdbc:postgresql://localhost:5432/${POSTGRES_DB}

Because Postgres will run on the EC2 host, choose one consistent production target.

Recommended production approach:
- If backend container uses host networking or host.docker.internal mapping, point to the EC2 host DB.
- Simpler alternative: keep Postgres in Docker too. But since you chose local Postgres for cost simplicity, explicitly configure the backend to use the host DB.

Production example env vars:
- SPRING_DATASOURCE_URL=jdbc:postgresql://172.17.0.1:5432/ballistics_db
  or another host-accessible address depending on Docker setup
- SPRING_DATASOURCE_USERNAME=postgres
- SPRING_DATASOURCE_PASSWORD=<strong_password>

Best practice:
- Make datasource URL fully env-driven in production.

2.5 Confirm Linux compatibility for renderer and ffmpeg flow

Backend render logic calls ffmpeg directly and also runs the native renderer.

Found in backend:
- RenderService.java invokes ffmpeg
- ray_tracer folder contains the native renderer source

Action:
- Build the renderer for Linux before deployment.
- Confirm the backend container includes the Linux renderer binary at a known path.
- Confirm ffmpeg exists inside the backend container.
- Confirm any native shared library dependencies are present.
- Do not rely on Mac-built binaries or Homebrew paths.

2.6 Google OAuth redirect URIs must be updated

Because localhost is currently used, Google login will fail in production unless updated.

Action:
- In Google Cloud console, add production authorized redirect URIs.
- Example:
  https://your-domain.com/login/oauth2/code/google
  or if backend stays on a separate host/path, the backend callback URL actually used by Spring Security
- Also update authorized JavaScript origins if required.

==================================================
3. Recommended production URL layout
==================================================

Simplest first deployment:
- Frontend: https://your-domain.com
- Backend proxied through nginx: https://your-domain.com/api
- OAuth/login endpoints also proxied through the same domain to backend

Why this is best for phase 1:
- No frontend/backend cross-origin split
- Fewer CORS problems
- Easier cookie/session handling if OAuth/session-based auth is used
- Cleaner nginx config

==================================================
4. EC2 setup steps
==================================================

4.1 Launch the instance
- AMI: Ubuntu 24.04 LTS or Amazon Linux 2023
- Size: t3.small minimum, t3.medium preferred
- Storage: start with 20-30 GB gp3 EBS
- Attach an IAM role with S3 permissions for your render bucket

4.2 Security group
Allow inbound:
- 22 from your IP only
- 80 from anywhere
- 443 from anywhere
Do NOT allow inbound:
- 5432 from the public internet
- 8080 from the public internet

4.3 Install base packages
- docker
- docker compose plugin
- nginx
- certbot if using Let's Encrypt
- postgresql server
- git (optional)

4.4 Enable services
- enable and start docker
- enable and start nginx
- enable and start postgresql

==================================================
5. Postgres on EC2 setup
==================================================

5.1 Install and initialize Postgres on the EC2 host
- create database: ballistics_db
- create strong postgres/app user credentials
- restrict listening address to localhost or private-only

5.2 Create schema
Your backend includes schema.sql in:
- backend/src/main/resources/schema.sql

Action:
- run that schema against the EC2 Postgres instance if Spring is not handling it automatically
- verify the jobs table and related fields are present

5.3 Lock down Postgres
- bind Postgres to 127.0.0.1 unless a different internal-only setup is required
- keep security group closed for 5432
- do not install pgAdmin on production unless needed temporarily

5.4 Backups
Because DB is local, backups are now your responsibility.

Set up:
- nightly pg_dump to a backup directory
- sync dumps to S3
- retention policy for old dumps

Minimum acceptable backup plan:
- one daily dump
- keep 7-14 days
- test restore once

==================================================
6. Backend container setup
==================================================

6.1 Create a production Dockerfile for the backend
The current repo contains:
- ballistics_backend/backend/Dockerfile

Verify it does all of the following:
- uses a Linux JRE/JDK base image
- copies the built Spring Boot jar
- installs ffmpeg
- copies the Linux renderer binary into the image
- exposes port 8080
- runs the jar

6.2 Backend runtime env vars
Use a production env file on EC2. Suggested variables:
- SPRING_PROFILES_ACTIVE=prod
- SPRING_DATASOURCE_URL=jdbc:postgresql://<host-accessible-db-address>:5432/ballistics_db
- SPRING_DATASOURCE_USERNAME=postgres
- SPRING_DATASOURCE_PASSWORD=<strong_password>
- POSTGRES_DB=ballistics_db
- APP_FRONTEND_URL=https://your-domain.com
- APP_ALLOWED_ORIGIN=https://your-domain.com
- GOOGLE_CLIENT_ID=<rotated_value>
- GOOGLE_CLIENT_SECRET=<rotated_value>
- APP_S3_BUCKET=<your_bucket_name>
- AWS_REGION=<your_region>

6.3 S3 auth
Preferred:
- use EC2 IAM role, not hardcoded AWS keys

Backend should be able to:
- upload rendered mp4 files to S3
- generate presigned download URLs if needed
- optionally delete old objects via lifecycle rules or app cleanup

6.4 Logging
- mount a host directory for container logs if needed
- use docker logs for first-pass debugging
- keep renderer stderr/stdout visible during setup

7. Frontend production setup
==================================================

7.1 Build Angular app for production
From ballistics_frontend:
- npm ci
- ng build

The build output should be copied to nginx's web root.

7.2 API base URL behavior
Frontend should call the backend through nginx using relative paths:
- /api/...
not:
- http://localhost:8080/...
- http://ec2-public-ip:8080/...

7.3 Login URL behavior
The current root .env has:
- login_url=http://localhost:8080/oauth2/authorization/google

For production, this should become same-origin, for example:
- /oauth2/authorization/google
or
- https://your-domain.com/oauth2/authorization/google

==================================================
8. nginx setup
==================================================

nginx should do two jobs:

1) serve Angular static files
- return index.html for SPA routes

2) reverse proxy backend routes
- /api -> backend container on localhost:8080
- /oauth2 -> backend container on localhost:8080
- /login -> backend container on localhost:8080
- /logout -> backend container on localhost:8080 if needed

Also configure:
- standard proxy headers
- larger client body size if render job payloads need it
- gzip optional

After DNS is set:
- add SSL certificate
- redirect HTTP to HTTPS

9. Deployment workflow on EC2
==================================================

Recommended simple workflow:

1. SSH into EC2
2. pull or copy app code to /opt/ballistics
3. build backend jar
4. build backend Docker image
5. build frontend static files
6. copy frontend build to nginx web root
7. restart backend container
8. reload nginx
9. run smoke tests

Suggested directories:
- /opt/ballistics/app
- /opt/ballistics/env
- /opt/ballistics/backups
- /var/www/ballistics for frontend build

12. Strong recommendation on deployment order
==================================================

Do the rollout in this order:

Phase A: local production-shape validation
- run Postgres locally
- run backend container against that DB
- verify Linux renderer + ffmpeg in container
- build frontend and test nginx-style relative /api behavior

Phase B: EC2 without domain first
- deploy on EC2 using public IP
- verify backend, DB, S3, and frontend all work

Phase C: domain + HTTPS + OAuth finalization
- point domain to EC2
- issue SSL cert
- update Google OAuth redirect/origin settings
- retest login/logout flow

13. Highest-risk items in this app specifically
==================================================

These are the most likely things to break first on EC2:

1. hardcoded localhost URLs in backend auth/CORS
2. frontend code or login flow still pointing to localhost
3. Linux renderer binary missing or incompatible
4. ffmpeg missing inside backend container
5. backend container unable to reach host Postgres
6. Google OAuth redirect URI mismatch
7. S3 permissions not available from the EC2 instance role

15. Bottom-line recommendation
==================================================

For your current priorities, the most practical first production deployment is:
- one EC2 instance
- local Postgres on the EC2 host
- Spring Boot backend in Docker
- Angular static frontend served by nginx
- S3 for rendered video files

That keeps monthly cost down while still giving you a real production deployment path.

Do not deploy the current codebase unchanged. The localhost/OAuth/CORS/secrets issues need to be fixed first.
