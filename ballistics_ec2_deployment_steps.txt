Ballistics app deployment plan for EC2
Date: 2026-04-01

Goal
Deploy the current Ballistics app to production on a single EC2 instance, with S3 for rendered videos and optional Route 53 for DNS. This is the simplest production path for the current codebase because the app already includes a Spring backend, Angular frontend, PostgreSQL, a native renderer binary, and ffmpeg-based video generation.

Recommended AWS services
1. EC2
   - Run nginx, the Spring backend, the Angular frontend build output, and PostgreSQL (initially).
   - Small starting point: t3.medium or t4g.medium if all binaries/images support ARM; otherwise t3.medium x86_64 is safer.

2. S3
   - Keep using S3 for rendered videos.
   - Keep lifecycle rule to delete objects daily.

3. IAM
   - Attach an instance role to EC2 with least-privilege S3 access to the render bucket.
   - This is better than storing AWS keys on the server.

4. Route 53 (optional but recommended)
   - Point your domain/subdomain at the EC2 public IP or Elastic IP.

5. Elastic IP (recommended)
   - Give the EC2 instance a stable public IP.

6. Google OAuth credentials
   - Update the Google OAuth app to use the production domain instead of localhost.

Services you can skip at first
- RDS: not required for the first EC2 deployment. Start with PostgreSQL on the same EC2 instance, then move to RDS later if needed.
- ALB/ACM: not required for a single-instance deployment. Simpler approach is nginx + Let's Encrypt on the instance.
- ECS/App Runner/Amplify: not needed if the decision is EC2.

What must be fixed in the code before deployment
The current app is still wired for local development in a few important places.

A. Backend has hardcoded localhost frontend URLs
Found in:
- ballistics_backend/backend/src/main/java/com/jgezalian/ballistics_backend/config/OAuth2Login.java
- controllers with @CrossOrigin annotations

Current hardcoded values include:
- http://localhost:4200
- defaultSuccessUrl("http://localhost:4200", true)
- logoutSuccessUrl("http://localhost:4200")

What to change:
- Replace those hardcoded URLs with environment-driven config, for example:
  - APP_FRONTEND_URL=https://ballistics.yourdomain.com
  - APP_ALLOWED_ORIGIN=https://ballistics.yourdomain.com
- Prefer a global CORS config over scattered @CrossOrigin annotations.

B. Frontend login URL is hardcoded to localhost:8080
Found in:
- ballistics_frontend/src/environments/environment.ts
- ballistics_frontend/src/environments/environment.development.ts

Current value:
- loginUrl: 'http://localhost:8080/oauth2/authorization/google'

What to change:
- In production, use a same-origin path such as:
  - loginUrl: '/oauth2/authorization/google'
- Keep API calls at:
  - baseApiUrl: '/api'

C. proxy.conf.json is only for Angular dev server
Found in:
- ballistics_frontend/proxy.conf.json

This is not production proxying. In production, nginx should serve the frontend and forward /api, /oauth2, /login, and /logout to the backend.

D. Renderer binary path is currently source-tree dependent
Found in:
- ballistics_backend/backend/src/main/java/com/jgezalian/ballistics_backend/service/RenderService.java

Current executable path:
- src/main/c/animate_sphere_on_plane

Problem:
- That path assumes the source tree exists on the production server in the same layout.

What to change:
- Move the renderer binary to a fixed runtime path on the server or inside the backend image/container, for example:
  - /opt/ballistics/bin/animate_sphere_on_plane
- Then load it from config, for example:
  - app.renderer.path=/opt/ballistics/bin/animate_sphere_on_plane

E. ffmpeg must exist on the production host
RenderService calls ffmpeg directly. That means the EC2 machine must have ffmpeg installed and available on PATH.

F. Database password and secrets handling need cleanup
Current compose file uses:
- db/password.txt
- pgAdmin with a plain-text password

For production:
- Do not expose pgAdmin publicly.
- Put secrets in an env file outside the repo or use SSM Parameter Store / Secrets Manager later.

Recommended production shape on EC2
Use same-origin deployment.

Browser requests:
- https://ballistics.yourdomain.com/          -> Angular static files from nginx
- https://ballistics.yourdomain.com/api/*    -> proxied by nginx to Spring Boot on localhost:8080
- https://ballistics.yourdomain.com/oauth2/* -> proxied by nginx to Spring Boot
- https://ballistics.yourdomain.com/login/*  -> proxied by nginx to Spring Boot
- https://ballistics.yourdomain.com/logout   -> proxied by nginx to Spring Boot

Why this is best for the current app
- Simplest OAuth callback setup
- Minimal CORS pain
- Frontend can keep using /api
- One public domain instead of split frontend/backend subdomains

High-level deployment steps
1. Create AWS infrastructure
2. Update the app config for production
3. Build the frontend, backend, and renderer
4. Provision the EC2 host
5. Install dependencies on EC2
6. Deploy PostgreSQL, backend, frontend, and nginx
7. Configure DNS + HTTPS
8. Update Google OAuth
9. Test end-to-end
10. Add hardening and backups

Detailed step-by-step plan

Step 1: Create the AWS pieces
1. Create or confirm the S3 bucket for rendered videos.
2. Create an IAM role for EC2 with permissions to:
   - s3:PutObject
   - s3:GetObject
   - s3:DeleteObject
   - s3:ListBucket (optional)
   on the specific render bucket.
3. Launch an EC2 instance.
   Recommended starter settings:
   - AMI: Ubuntu 24.04 LTS
   - Instance type: t3.medium
   - Storage: 30-50 GB gp3
   - Security group inbound:
     - 22 from your IP only
     - 80 from 0.0.0.0/0
     - 443 from 0.0.0.0/0
   - Attach the IAM role.
4. Allocate and attach an Elastic IP.
5. If using a domain, create the DNS record in Route 53 or your current DNS provider.

Step 2: Decide what runs where
Initial deployment on one EC2 instance:
- nginx
- Angular production build files
- Spring Boot backend
- PostgreSQL
- ffmpeg
- native renderer binary

Optional later improvements:
- Move PostgreSQL to RDS
- Move static frontend to S3 + CloudFront
- Put backend in Docker Compose or ECS

Step 3: Update backend configuration
Create production-configurable properties.
Suggested environment variables:
- SPRING_PROFILES_ACTIVE=prod
- POSTGRES_DB=ballistics_db
- POSTGRES_PASSWORD=...
- GOOGLE_CLIENT_ID=...
- GOOGLE_CLIENT_SECRET=...
- APP_S3_BUCKET=your-bucket-name
- APP_TMP_DIR=/var/lib/ballistics/tmp
- APP_FRONTEND_URL=https://ballistics.yourdomain.com
- APP_ALLOWED_ORIGIN=https://ballistics.yourdomain.com
- APP_RENDERER_PATH=/opt/ballistics/bin/animate_sphere_on_plane

Backend changes to make:
1. Replace hardcoded localhost URLs in OAuth2Login.java with config values.
2. Replace @CrossOrigin("http://localhost:4200") with config-driven CORS or remove them if same-origin proxying is used.
3. Replace the hardcoded renderer path in RenderService with APP_RENDERER_PATH.
4. Change app.s3.bucket in application.properties to use an env var, for example:
   app.s3.bucket=${APP_S3_BUCKET}
5. Keep app.tmp.dir configurable and point it to a durable writable folder on EC2.

Step 4: Update frontend configuration
Use same-origin production values.
Suggested production values:
- loginUrl: '/oauth2/authorization/google'
- baseApiUrl: '/api'

Notes:
- The frontend should not point to localhost in production.
- The Angular dev proxy file is irrelevant once nginx is in front.

Step 5: Build artifacts locally or in CI
Build the renderer:
- Build the C/C++ renderer in release mode.
- Confirm the final executable runs on the same OS/architecture you will use on EC2.

Build the backend:
- mvn clean package
- Output should be the Spring Boot jar.

Build the frontend:
- npm ci
- npm run build
- Output should be the Angular dist folder.

Important architecture note:
- If you build the native renderer on macOS, do not copy that binary to Ubuntu EC2.
- Build the renderer for Linux on Linux, or compile it directly on the EC2 host.

Step 6: Prepare the EC2 machine
SSH in and install packages:
- nginx
- openjdk 25 runtime (or align the app to a more common LTS JDK such as 21)
- postgresql
- ffmpeg
- build-essential / cmake if compiling the renderer on the box
- certbot + python3-certbot-nginx

Also create directories such as:
- /opt/ballistics/backend
- /opt/ballistics/frontend
- /opt/ballistics/bin
- /var/lib/ballistics/tmp
- /var/log/ballistics

Step 7: Install and configure PostgreSQL
Initial simple path:
- Run PostgreSQL on the same EC2 instance.
- Create:
  - database: ballistics_db
  - user: postgres (or a dedicated app user)
  - strong password
- Confirm the backend schema initialization works.

Later upgrade path:
- Move DB to RDS when you want easier backups, patching, and durability.

Step 8: Deploy the backend
Two valid ways:

Option A: systemd service directly on the host
- Copy the jar to /opt/ballistics/backend/
- Put env vars in a root-readable env file, for example:
  - /etc/ballistics/ballistics.env
- Create a systemd service that starts the Spring Boot jar on port 8080.

Option B: Docker Compose on the EC2 instance
- Containerize backend and database.
- Keep nginx on the host.

For your current app, Option A is the fastest first deployment unless you specifically want to dockerize everything.

Step 9: Deploy the renderer binary
- Copy or build the Linux renderer binary to:
  - /opt/ballistics/bin/animate_sphere_on_plane
- Make it executable:
  - chmod +x /opt/ballistics/bin/animate_sphere_on_plane
- Test it manually with sample args.

Step 10: Deploy the frontend
- Copy the Angular dist output into a directory nginx can serve, for example:
  - /var/www/ballistics
- Configure nginx to serve index.html and support SPA routing.

Step 11: Configure nginx reverse proxy
nginx should:
1. Serve the Angular build at /
2. Proxy the following to http://127.0.0.1:8080:
   - /api/
   - /oauth2/
   - /login/
   - /logout
3. Forward cookies and headers correctly
4. Support large enough request bodies if needed

Important note about your current backend routes:
- The frontend expects API calls under /api.
- The current Spring controllers may not all be prefixed with /api.
- You may need either:
  a. nginx path rewriting from /api/* to /* on the backend, or
  b. change Spring controller mappings to include /api.
- Pick one convention and keep it consistent.

Step 12: Configure HTTPS
Simplest single-instance setup:
- Point the domain to the EC2 Elastic IP.
- Use certbot with nginx to issue a Let's Encrypt certificate.
- Force HTTP -> HTTPS redirect.

Step 13: Update Google OAuth settings
In Google Cloud console, update the OAuth client.
You will need:
- Authorized JavaScript origins:
  - https://ballistics.yourdomain.com
- Authorized redirect URIs:
  - https://ballistics.yourdomain.com/login/oauth2/code/google

Also ensure the backend success and logout URLs point at the production frontend URL.

Step 14: Validate S3 access from the instance
Because the backend uses the AWS SDK default credential chain, attaching an EC2 IAM role should be enough.
Test:
- create a render job
- confirm upload to S3 works
- confirm presigned URL generation works
- confirm lifecycle deletion rule still removes objects daily

Step 15: End-to-end production test checklist
1. Open the site over HTTPS.
2. Confirm Angular app loads.
3. Click Google login.
4. Confirm OAuth redirects back to the site.
5. Create a render job.
6. Confirm backend launches renderer.
7. Confirm ffmpeg creates mp4.
8. Confirm mp4 uploads to S3.
9. Confirm the frontend can list and play/download the render.
10. Confirm delete works and also removes the S3 object.
11. Restart the instance and confirm services auto-start.

Step 16: Hardening after the first successful deploy
1. Restrict SSH to your IP only.
2. Do not expose PostgreSQL publicly.
3. Do not expose pgAdmin publicly.
4. Run the backend under a dedicated Linux user.
5. Move secrets out of repo files.
6. Add automatic security updates.
7. Add CloudWatch agent or another log shipping setup.
8. Add PostgreSQL backups.
9. Add a basic uptime monitor.

Recommended order of work from here
1. Fix production config in backend and frontend.
2. Remove hardcoded localhost URLs.
3. Make renderer path configurable.
4. Confirm Linux build of the renderer.
5. Build frontend and backend artifacts.
6. Launch EC2 and attach IAM role.
7. Install nginx, Java, PostgreSQL, and ffmpeg.
8. Deploy app on same-origin setup.
9. Add domain + HTTPS.
10. Update Google OAuth production settings.
11. Run full end-to-end test.

Strong recommendation for this codebase
For the very first production deployment, use:
- one EC2 instance
- nginx on the host
- Spring Boot on the host via systemd
- PostgreSQL on the same EC2 instance
- S3 for video storage
- IAM role for bucket access
- Let's Encrypt for TLS

Then, after the app is stable, consider phase 2:
- move PostgreSQL to RDS
- optionally dockerize the runtime more cleanly
- optionally move frontend to a CDN/static host

Known app-specific issues discovered in the current code snapshot
1. Hardcoded localhost in backend CORS/OAuth config
2. Hardcoded localhost login URL in frontend env files
3. Angular proxy file is dev-only, not production routing
4. Renderer path depends on source-tree layout
5. ffmpeg is required on the production server
6. Dockerfile shown in the backend currently does not install ffmpeg or package the native renderer
7. Current setup appears closer to local development than production, so expect a config pass before deployment
